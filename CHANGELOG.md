## Patch 2026-07-20t — "The Minute-Long Hiccup"

The strategy kept printing `ML POST FAILED … timed out` — lots of them. Every so often the AI service simply stopped answering for a few seconds, and anything the strategy tried to send in that window was dropped on the floor. It turned out the service was spending most of its life re-reading the same enormous files over and over, for the benefit of a few dashboard panels.

### 🐛 Fixed
- **The AI service no longer freezes.** A plain "are you alive?" request used to take anywhere up to **8.7 seconds** during these hiccups. It now answers in **3–25 milliseconds**, with the internal freeze counter sitting at a flat zero.
- **Four separate "cheap-looking" jobs were each reading a whole file every time they ran.** The worst offender: every time the strategy logged a single exit sample, the service re-read the *entire* 30 MB sample file — just to check the first line was still labelled correctly. That happened hundreds of times a minute. Each one is now either reading one line, or remembering the answer until the file actually changes.
- **The dashboard's verification panel was the "every minute or two" culprit.** Once a minute it re-parsed a 265 MB file inside the service, locking everything else out for roughly fifteen seconds. That work now happens in a separate background process, so the panel still refreshes and nothing else notices.
- **Restarting the service actually restarts it now.** The restart helper had been accidentally killing *itself* partway through — it never got as far as starting the new copy, so the service stayed down for up to two minutes until a watchdog noticed. Restarts now complete in about five seconds.

### 🛠️ Under the hood
- The background job that feeds the model-health panels was re-reading roughly **695 MB every ten seconds**. The dashboard-only parts of it now run once every ten minutes; nothing they display changes faster than that anyway.
- Added a permanent diagnostics page for this class of problem. It watches the service's own responsiveness and, the moment it freezes, snapshots what every thread inside it is doing — which is how all four causes were named instead of guessed at.

### 📋 Checked and dropped
- **"Turn off virus scanning for the trading folder" — not needed.** The first draft of these notes suggested it, on the basis of Windows Defender looking busy at the time. Measured properly afterwards: reading that 265 MB data file runs at **2.8 GB/s**, the same speed as a file Defender has no reason to watch. There is no scanning penalty to remove. The slow part was never the disk — it was the maths.
- **"A stray monitoring script is eating a CPU core" — also not true.** It looked that way in one performance snapshot, but Windows was attributing the wrong number to it: measured directly, that script uses **0 milliseconds of CPU** over five seconds and does 0.13 seconds of work per fifteen-second check. The only meaningful CPU user on the machine is NinjaTrader itself, which is expected.

*Dev note: symptom was a trivial in-memory `/health` taking 0.5–8.7s in ~20s bursts while every endpoint measured in isolation was fast. Ruled out by measurement, in order: request payload size (a 844 KB `/predict-exit` body costs 82ms), garbage collection (`gc.callbacks` recorded zero pauses ≥300ms across 5.5s loop stalls), TIME_WAIT/port exhaustion, box-wide contention (8766 probed in lockstep stayed at 2–200ms), and request flooding (44 concurrent connections, no stall). The real answer came from a stall sampler thread dumping every thread's frames while the event loop was stuck: the loop was sitting in trivial socket calls **waiting for the GIL**, held in turn by `known_groups()` globbing the NT8 root on the event loop, `create_exit_tsv_if_missing()`'s `readlines()`, `phase3_unlocked_for_group()`'s per-call history parse, and `_compute_verification_readiness()`'s full 265 MB in-process scan. Fixes: mtime-keyed caches, first-line-only header check, assembly moved off the loop via `run_in_threadpool`, readiness scan moved into the existing `ProcessPoolExecutor`, a 5s TTL on `/model-health`, `async def health`, and a heavy/light split in the refresh cadence (60s / 600s).*

## Patch 2026-07-20k — "The Blinking Terminal"

A little black terminal window kept flashing onto the desktop about once a minute — startling, but harmless once we traced it. Two of the background "watchdog" helpers (the ones that keep the ML service and NinjaTrader itself alive) were being launched in a way that briefly showed a command window every single time they ran. Every other watchdog on the machine already runs completely invisibly; these two just missed the memo.

### 🐛 Fixed
- **The watchdogs now run silently.** Rerouted the offenders — the ML-service watchdog (fires every minute) and the NinjaTrader process watchdog (every two minutes) — through the same invisible-launcher trick the other background jobs already use. No more window flashing onto your screen while you work.
- **Killed the *persistent* mystery window too.** On a second look there was a black terminal window that had been sitting open since sign-in — not flashing, just *there*. It turned out to be the little helper that watches this patch-notes repo and auto-publishes changes. It runs a never-ending loop, and Windows insisted on giving it a visible window even though it was told to hide. Rehosted it in a truly windowless way; the window is gone and stays gone across restarts.

### 🛠️ Under the hood
- Added two tiny hidden-launcher scripts for the quick watchdogs and pointed their scheduled tasks at them instead of calling PowerShell directly (which is what caused the flash). The watchdogs themselves are unchanged — same job, same one- and two-minute schedules, they just don't show their face anymore. A one-time admin approval was needed; a re-apply script is saved alongside the maintenance tools in case a task ever gets rebuilt.
- The persistent one needed a *different* fix (see dev note) — a fire-and-forget launcher would have quietly spawned a fresh copy of the watcher every single minute and piled up dozens of them. Now there's exactly one, and it's invisible.

### 🔁 Follow-up (same day)
- **Windows were still appearing, so every background job got the same treatment.** Rather than fix them one at a time as they showed up, all eight scheduled jobs that can open a console — the auto-commit (runs every 5 minutes), both service watchdogs, the trend watchdog, the off-site backup upload, the log cleanup, the weights backup, and the patch-notes publisher — now launch in a genuinely windowless mode. The prime remaining suspect was the 5-minute auto-commit: it *claimed* to be hidden, but "hidden" doesn't stop Windows from handing it a visible window. Confirmed with a two-minute watch across every job's firing cycle: **zero windows.**

*Dev note: two different problems, two different fixes. (1) `-WindowStyle Hidden` on a scheduled `powershell.exe` still blinks a console for a frame; the reliable fix for short quick-fire tasks is a `wscript.exe` + VBS `Run …, 0` wrapper (proven by the MEGA-backup task). (2) The park autopush watcher is a long-running `while($true)` loop, and Windows Terminal — the Win11 default terminal — hijacks the console handoff and shows a window despite `Hidden`. `wscript` is wrong for a loop: it exits instantly, so the task's 1-min re-trigger defeats `IgnoreNew` and piles up watchers. Fixed by launching it as `conhost.exe --headless powershell … -File watch_and_autopush.ps1` — `--headless` = no window and no Terminal handoff, and conhost stays alive as the task process so `IgnoreNew` still holds it to one instance. All applied via `Maintenance\make_watchdogs_invisible.ps1` (idempotent, logs a transcript).*

## Patch 2026-07-20n — "Sleep On It First"

The auto-tuner's problem was never how big its changes were — it was how *often*. It could nudge the same setting every five minutes, walking it clear across its range before anyone noticed. Today it learns patience, and we wipe the slate clean so it starts fresh from the designed defaults.

### 🆕 New
- **A change now has to prove itself before it happens.** The tuner must want the exact same adjustment for **three runs in a row (~15 minutes)** before it's allowed to make it. A one-off blip in the data no longer moves anything — only a trend that actually persists.
- **After it changes something, that setting is frozen for 6 hours.** This forces the tuner to live with its decision and watch how it plays out before touching the same dial again, instead of re-deriving a brand-new target every five minutes.

### ♻️ Reset
- **Wiped every auto-tuned value back to its designed default to start fresh** — entry filters, expiry, slippage cushion, pullback distances, position sizing, all of it. Paired with a clean cutoff: the tuner now ignores every trade recorded before this reset, so it can't immediately re-derive the same drift from the old, messy history. The very first run after the reset correctly found nothing to do.

### ⚠️ Known issues
- The live tuner clobbered one pullback value **twice more** while I was still saving the new code — each 5-minute run that fired before the patch landed re-applied the old behavior. Once the patience/cooldown logic was actually on disk, the next run went quiet and the resets held. Lesson logged: land the guard code *first*, reset values *second*.

*Dev note: both filters live in `_try_apply`, the single chokepoint all six apply-sites funnel through — persistence streak + cooldown stamp per constant in a new `auto_apply_change_state.json`, keyed by the patch description. `PERSISTENCE_RUNS=3`, `COOLDOWN_HOURS=6`. Fresh-start floor is `EVIDENCE_RESET_ISO=2026-07-20T18:02:00`, applied via a `_FloorDict` wrapper on `_evidence_cutoffs` so it covers constants with no prior history too. Unit-tested (persistence blocks runs 1–2 / applies run 3; cooldown holds 6h; absent-run resets streak); a live `--apply` returned "Nothing to do." NQ Tier-2 risk target reset 877.06 → 1000.*

After fixing the one-tick pullback, I checked every other knob the auto-tuner is allowed to touch — and the pullback wasn't alone. The same July-19 evidence storm had shoved *everything* to an extreme at once: the entry filters were flung wide open, and the safety cushion we set aside for entry slippage had been shaved down to almost nothing. None of these were random garbage like the pullback was, but all were parked at the far edge of what's allowed, which is its own kind of broken.

### ⚖️ Balance
- **Entry filters reset to their intended strictness.** The three gates that decide whether a setup is worth taking (two momentum gates and a stochastic one) had all been loosened to their maximum, meaning the strategy was waving through setups it would normally skip. Reset all three back to the designed baseline (no extra loosening).
- **Restored the entry-slippage cushion.** The buffer we reserve to absorb slippage on entries had been ground down from its designed 10% to about 2.4% — almost no cushion. Put it back to 10%.

### 🛠️ Under the hood
- **Capped how far the auto-tuner can open the gates.** Same lesson as the pullback and expiry fixes: resetting isn't enough if the automation can just re-inflate. Lowered the gate-loosening ceiling to a third of what it was, so even at full stretch the filters can't be flung wide open again.
- **Re-seeded the tuner's evidence clock** on every value I reset, so it has to re-earn any future change from scratch rather than snapping back on day-old data.

### ⚠️ Known issues
- **The auto-tuner runs every 5 minutes and edited the strategy mid-session while I was working** — it briefly reverted one pullback value before the new floor was in place; I put it back. It can't be paused without an admin action, so it was left running. The pullback, expiry, and gate limits are now hard walls it can't cross; the slippage cushion is protected only by the re-seeded evidence clock, so keep half an eye on it over the next few days.

*Dev note: the runaway is `TemaLimitAutoApplySizing` (SYSTEM scheduled task, 5-min cadence). Guards now in place: `MinPullbackTicks=5`, `EXPIRE_MAX_TOTAL_EXTRA=7`, `GATE_MAX_TOTAL_WIDEN` 15/15/0.15 → 5/5/0.05. Slippage floor left at 0.02 per call — cushion reset but not hard-guarded. Post-guard 5-min runs (17:23–17:43) all applied nothing.*

## Patch 2026-07-20j — "One Tick Is Not a Pullback"

Caught a bad one. Most of the entry templates were trying to buy dips and sell rips with a "pullback" of a **single tick** — the smallest price increment there is. A pullback entry is supposed to wait for the price to come back a meaningful distance before jumping in; a one-tick wait is basically no wait at all, and we'd already learned (back on July 18) that trading this way *lost money on three of the four instruments*. We fixed it then. It quietly came back.

### 🐛 Fixed
- **Restored real pullback distances on every high-tier template (and all the small-cap/RTY ones).** Each instrument scales its pullback by a tuning multiplier; five of those multipliers had been ground down to near-zero (around one-hundredth of their intended size), which collapsed the actual pullback to a single tick across a huge swath of templates. Put all five back to their designed values, so pullbacks are once again measured in the *tens* of ticks where they belong.
- **Found out why it came back — and slammed the door.** The auto-tuner that adjusts pullback distance based on missed fills had no lower limit: every time an order just missed, it nudged the target *tighter*, and with nothing stopping it, it kept going all the way down to one tick. That's the loop that undid July 18's fix. It can't happen again now (see below).

### 🛠️ Under the hood
- **Added a hard floor of 5 ticks on any pullback, enforced in three places.** The strategy itself now refuses to place a pullback tighter than 5 ticks no matter what the multiplier or volatility adjustment says, and the auto-tuner is forbidden from even *proposing* anything tighter than 5. Five ticks is the strategy's own designed minimum (what the calmest low-tier templates already use), so this only cuts off the broken range, nothing legitimate.

*Dev note: this is the same regression as the July-18 "1-tick high-tier" fix, which had relied on a soft "seeded cutoff" that fresh no-fill evidence could out-vote over time. This time the floor is a real constant (`MinPullbackTicks` in the strategy, mirrored by `MIN_PULLBACK_TICKS_FLOOR` in the automation), not a nudge that can be eroded. Strategy side autocompiles; the automation floor applies on its next run (fresh import, no restart).*

## Patch 2026-07-20i — "The Order That Overstayed Its Welcome"

Noticed the entry orders were hanging around far too long. When the strategy wants in on a trade, it places a limit order at a specific price and waits for the market to come to it — but if it never does, that order should give up and cancel so a stale price doesn't fill minutes later when the setup has gone. That "give up" timer had quietly ballooned to **22–26 minutes** on the tighter templates. A limit order resting live for nearly half an hour is asking to get filled at exactly the wrong moment.

### ⚖️ Balance
- **Entry orders now cancel after 2–6 minutes instead of 22–26.** The timer starts from a small base (2–6 min) and the auto-tuner is allowed to add extra patience on top when the evidence says orders are just missing the fill. That "extra" had been cranked all the way up to +20 minutes. Reset it to zero, so expiry drops straight back to the intended 2–6 minute floor.

### 🛠️ Under the hood
- **Capped how far the auto-tuner can push the timer back up.** The same automation that inflated it would happily do so again within a few daily runs — it was only limited by a ceiling of +24 minutes. Lowered that ceiling to +7, so even at full stretch the timer can't exceed ~9–13 minutes. The reset is now durable, not a one-day reprieve.

*Dev note: two knobs, two files. The reset lives in the strategy (`EntryExpireExtraMinutes*` → 0) and autocompiles. The ceiling (`EXPIRE_MAX_TOTAL_EXTRA` 24 → 7) lives in the dashboard/automation code and applies on the auto-tuner's next run — it re-imports fresh each run, so no restart needed. Base timer itself is untouched: still 2–6 min scaled by template selectivity.*

## Patch 2026-07-20h — "Sweeping Up the Trades That Never Said Goodbye"

The verification suite was flagging failures, so I dug in — and two of them turned out to be worth fixing in very different ways. One was real data damage to clean up; the other was a health check counting wrong. Earlier today's fix stopped the strategy from *creating* trades that forget to record their exit — but it couldn't go back and fix the ones already broken. Thirteen trades from the last day and a half had closed without ever writing their "I exited here" marker, leaving **22,651 rows** of training data that quietly teach the model a lie: that positions never get out. This patch cleans those up, and separately teaches the "label drift" check to stop raising false alarms.

### 🐛 Fixed
- **Removed 13 "exitless" trades that were poisoning the exit model's training data.** Each one was a real, closed trade whose final exit was never recorded (all closed *before* today's earlier fix went in), so its entire history read as one long "still holding" with no ending. Left in place, they'd bias the model toward holding trades too long. They're not deleted — every row was moved into a dated archive folder first, so the evidence is kept. The failing health check went straight back to green afterward.
- **A second health check was crying wolf.** The "label drift" check watches whether the mix of long/short/no-trade signals shifts suddenly (a sign something broke in labeling). It was flagging a false alarm on the 1-minute ES data. The cause: when the strategy shadow-tests a setup across all 40 templates, it writes one near-identical row per template — so a single real setup can look like 40. The check was counting all of them, while the model that actually trains on this data throws the duplicates away first. Two long setups looked like a flood of longs. The check now counts the same de-duplicated way the trainer does; the false alarm is gone (it drops to a mild, honest "heads up" instead), and this fixed the same over-counting for every other symbol too.

### 🛠️ Under the hood
- New cleanup tool picks trades to quarantine by the *exact same rule* the health check uses (closed, and no recorded exit), rather than a hardcoded date — so it stays correct if this ever recurs. It refuses to touch anything that might still be an open position (30-minute safety margin), archives before it removes, and preserves any rows the live service writes while it runs.
- The label-drift fix reuses the de-duplication logic the duplicate-scan check already had, so the two checks now agree on what "the data" means. Each result also now reports both its raw and de-duplicated row counts so the gap is visible at a glance.

### ⚠️ Known issues
- One lookalike trade in a different data series shares an ID with a cleaned one (IDs are timestamp + symbol, so two series can collide). It also has no recorded exit but is still inside the "might be open" window, so it was deliberately left alone — it'll either record its exit normally or get caught on a later check.

*Dev note: the exit-model root cause and the strategy-side fix are in patch 20-07-20e / the midday guide entry; the cleanup half of this patch is purely for the orphans that predate it (that check reads the files directly, so it cleared with no restart). The label-drift fix is a code change to the health service, so it needs an 8765 restart before the live dashboard shows the corrected verdict — validated out-of-process against live data in the meantime.*

## Patch 2026-07-20h — "The Alert That Shows Its Work"

Phone alerts for the live account used to say *what* was ordered but never *why*. Now a resting entry order arrives with the full settings profile that produced it — so you can judge a trade from your pocket without opening the platform.

### 🆕 New
- **Pending-order alerts now carry the template that placed them.** The alert appends the active template's number, its selectivity, the pullback distance, and the three indicator gates (money-flow, strength, stochastic), plus that market's risk-per-trade and daily limits. Entry orders only — protective stops move constantly as they trail, and repeating five lines on every step would bury the phone. The quantity line was dropped as redundant.
- **The pullback shows both numbers when they differ.** The template proposes a distance; live volatility stretches or compresses it before the order goes out. You now see both — the setting and what actually priced the order — rather than a number that quietly wasn't used.
- **Machine-learning trades announce themselves.** When the model actually drives an entry, one extra line shows its confidence against the minimum it had to clear. It uses the *same* test the dashboard uses, so the phone alert and the dashboard's direction breakdown can never disagree about what counts as an ML trade.

### 🛠️ Considered and deliberately rejected
- **Renaming order signals to tag ML trades.** The obvious idea — call it `YM_MLL` instead of `YM_L` — turns out to be genuinely dangerous. The strategy recognises its own entries from a fixed list of names; a name missing from that list means the code that arms the protective stop never runs. Every ML trade would open **with no stop**, and its exits would be silently ignored. It was also unnecessary: the dashboard already separates ML entries into their own rows.
  *Dev note: this is the same failure mode as this morning's naked position, which is exactly why it was worth stopping to trace before writing anything.*

### ℹ️ Worth knowing
- The ML rows on the dashboard look empty because the models **aren't trained yet** — the entry model has driven 4 trades out of ~445, and the ML exit has never fired once. That's expected, not broken (earlier firings were poisoned data, since purged). Both dashboard cards fill themselves in automatically once the models go live; nothing further to build.

## Patch 2026-07-20f — "A Stop With Nothing to Stop"

When the machine came back after its restart, NinjaTrader tidied up the frozen positions on its own — and closed the stuck NQ short at a genuine **+$4,340**. But it left one thing behind: the stop-loss placed by hand earlier was still sitting in the market with no trade behind it, quietly following the price down. A stop like that isn't protecting anything — if price had ticked back up to it, it would have *opened a brand-new trade* out of nowhere. The safety alarm never noticed, because it was only ever taught to look for the opposite problem.

### 🆕 New
- **The safety monitor now watches for abandoned stops, not just unguarded trades.** Until today it asked one question every minute: "is there a position with no stop?" Now it also asks the mirror question: "is there a stop with no position?" — and if it finds one, it names the exact order to cancel and warns which direction it would accidentally open. It caught the real one on its very first run.

### 🐛 Fixed
- **The trade journal had the NQ trade recorded as a +$80 win. It was actually +$4,340.** During the freeze, a phantom "close" was logged that never really happened, so the journal booked a tiny scratch while the real position stayed open and was closed much later for a large profit. Corrected the entry to the real close, added a sibling trade the journal had missed entirely, and removed two duplicate "manual close" rows that were echoes of trades already recorded. (Backed up first; this is a simulated account, so it's bookkeeping accuracy, not real money.)

### ✅ Reconciled
- **Checked the live trade journal against NinjaTrader's own fill record for the whole day.** The **real-money account matched exactly.** All the drift was on the practice accounts, and every bit of it traces back to today's freeze-and-restart. A subtle trap worth flagging: the *totals* matched perfectly, because a phantom extra trade and a missing trade happened to cancel out — a reminder that "the numbers add up" is not the same as "the numbers are right."

*Dev note: I also fumbled and un-fumbled a detail mid-repair — briefly "fixed" the file's line endings in the wrong direction before checking which way they actually went, then put it back. No harm done, but worth owning. ## Patch 2026-07-20e — "The Robot That Froze With Its Guard Down"

A live short on the real-money account was left without a stop-loss for over half an hour, and the robot that was supposed to be watching it had quietly frozen solid. ## Patch 2026-07-20d — "The Exits That Never Got Written Down"

The exit robot has been learning from trades whose endings were missing. Not corrupted — missing. Yesterday's fix caught most of them; today's found the rest, and corrected a wrong diagnosis of my own along the way.

### 🐛 Fixed
- **Trades closed by hand never told the learning system they'd ended.** When a position is flattened manually — or by NinjaTrader itself, or recovered after a crash — the trade is recorded properly in the P&L, the dashboard and the template ledger. Every one of those closes left behind a long transcript reading "still holding, still holding, still holding" that simply stops, with no ending. Thirteen trades closed that way in a single day. The exit robot was being taught, quite literally, that positions never close. It now records the ending on those closes too.

## Patch 2026-07-20 — "Dead Man's Switch"

Two ES short trades filled into a fast spike and then ran with **no stop loss at all**. Three separate safety nets were supposed to catch that. All three were broken, in three different ways. This patch fixes all three and adds a fourth that checks every tick.

### 🐛 Fixed
- **The trailing stop was barely working on three of the four markets.** As a trade moves in your favour, the strategy is supposed to drag the stop up behind it, locking in more profit the further it runs. It does that properly on the Nasdaq — the market the chart is actually set to — where it adjusted the stop 164 times in one hour this morning. On the S&P, the Dow and the Russell, which are attached to the same chart as extra feeds, it managed **2 and 11 adjustments over the same hour and a comparable number of trades**. Those three markets have effectively been trading with a stop that gets set once early and then sits there. It now updates on every price tick for every market, not just the chart's own.
  *Dev note: the interesting part is that the strategy was not failing to notice the run-up — it was noticing too late to act. It measures the best price reached by reading the high of the current bar, which is correct, but on an attached feed it only looks when that bar finishes. By then price has usually come back off the high, so the stop level it now wants to set sits above where the market currently is. A safety check added yesterday refuses to place a stop above the market (it was added for good reason — placing one there fires it instantly). So the request gets thrown away entirely and the stop stays exactly where it was. Late measurement plus a correct safety check equals a stop that never moves at all. Trailing on ticks fixes it by setting each new level on the way up, while that level is still below the market and therefore legal.*
  *Worked example — this is the trade that exposed it. A Dow long this morning entered at 52198 and ran 65 points in your favour, a peak of $325. The stop moved once, four minutes in, to lock in $30, and then never moved again for the next ten minutes while all of that run happened. It closed for $90. Had the stop kept pace it would have been sitting at $172 by the peak. The $90 was itself lucky — the stop was sitting at a level worth $30 and happened to fill twelve ticks better on a gap.*
  *Honesty note on the evidence, because it cuts both ways: this could not be properly A/B tested. Replaying the day needs tick-by-tick market data that either isn't downloaded or doesn't cover the window, and a standard backtest can't reproduce the fault at all. Comparing the Nasdaq (where trailing works) against the other three across 190 trades shows **no difference at all** in how much of a winning move gets captured — 42.3% versus 42.5%. Only when you narrow to the big runs, where a trailing stop is actually supposed to earn its keep, does the Nasdaq pull ahead (69% versus 58%). That's a cut chosen after seeing the flat headline number, on 49 trades, comparing different markets — suggestive, not proof. The mechanical fault is beyond doubt; that fixing it makes money is not. Worth watching rather than assuming.*
- **Every stop-out was being recorded twice.** Whenever a trade closed at its stop, the strategy logged it once correctly and then, on the very next bar, logged it *again* as if you had manually closed the position by hand. You hadn't. 42 duplicate trades had piled up since the feature went live yesterday evening, inventing **$3,106.25 of profit that never happened** across nine accounts — and feeding those made-up numbers into both the daily loss cut-out and the template scoreboard that decides which settings to trade next.
  *Dev note: the strategy keeps a backup record of the open trade in case it loses track of it. When a stop fills, that backup is deliberately left in place and cleared a moment later. The problem was where "a moment later" sat — the code that writes the dashboard status ran first and read the backup, saw a trade that the broker had already closed, and concluded someone must have closed it by hand. The duplicate-protection that was supposed to catch exactly this only worked in one direction: it covered "closed by hand, then the fill arrives", never "the fill arrives, then the stale backup is read". Fixed by reconciling the backup before anything reads it, and only when the account is genuinely flat — that last condition is what keeps it from interfering with a trade recovered after a restart, where the backup is the only record that exists.*
- **The first attempt at that fix only covered half the problem.** It corrected the check that runs when a new price bar forms — but the strategy also refreshes its position display every 5 seconds off live price ticks, precisely so that charts which only update on a completed bar don't go stale in between. That second path had the same flaw and hadn't been touched. On a one-minute chart the 5-second refresh gets there first, so the fake trade was still recorded.
  *Dev note: the evidence was in which trades survived. After the first fix, two stop-outs on fast tick-based charts came through completely clean, while one on a one-minute chart still produced a duplicate — its stop filled at 08:05:38 and the next bar wasn't until 08:06:00, leaving a 22-second window the 5-second refresh walked straight into. That split is the useful signal for next time: duplicates appearing ONLY on slower chart types point at a refresh path rather than the bar path. The reconciliation is now a single shared routine called from all three places that read the position, so a future fourth reader is the only way this comes back.*
- **Verified against NinjaTrader's own execution list rather than assumed.** For one account across an eleven-hour window, NinjaTrader recorded 15 exits. All 15 matched the trade log exactly, to the second and to the tick. The 11 extra "closed by hand" entries matched nothing at all — 15 real fills, 26 logged trades.
  *Dev note: worth writing down because the duplicates do not look like duplicates. Only the entry details match; the exit time, exit price, profit and exit reason all differ, so a pair reads as two separate trades at a glance. Worse, the fake row is stamped with the bar's time rather than the moment it was written, so on tick and line-break charts it can appear to happen an hour BEFORE the real exit it duplicates. Sorting by time actively misleads here.*

- **The trend dashboard's new "is it alive" alarm cried wolf on the Nasdaq within hours of being installed.** Last night the alarm was rewritten to watch each robot's heartbeat — a marker written every time it processes a bar — rather than watching for prediction requests, which only happen when a setup forms and so went quiet on a normal thin Sunday. That part was right and stands. What was wrong was the deadline attached to it: a single flat two hours for every market. This morning it declared the Nasdaq silent for 312 minutes. It wasn't broken. NinjaTrader's own log shows the robot running without interruption overnight, and inside that stretch the Nasdaq went **281 minutes between heartbeats while perfectly healthy** — because its bars are built from order-flow imbalance, and a balanced overnight market simply doesn't produce them. Meanwhile the euro was writing one every three to six minutes. No single deadline can serve both: anything patient enough for the Nasdaq is asleep through a euro outage lasting hours. Each market now learns its own normal rhythm and is judged against that — the alarm sounds at three times an instrument's own worst typical gap, never sooner than two hours. Until a market has shown enough of its rhythm to characterise it, it gets a deliberately generous eight-hour benefit of the doubt rather than a guess. The warning text now also states which deadline it used and whether it was learned or provisional, so the next alert can be judged without opening the code.
  *Dev note: the wrong turn is worth recording. The first diagnosis argued the Nasdaq must be broken because its order-flow setting (400) is smaller than the S&P's (500), so it ought to produce bars more often, not less. That comparison is meaningless — those numbers sit on different scales on purpose, which is exactly why each market gets its own, and the code says so in a comment right at the top of the file. Reading it as a like-for-like ratio manufactured a fault out of nothing and sent a real outage investigation down a blind alley. The rule to keep: judge a market against its own history, never against another market's settings. User caught this one.*
  *Second dev note, unrelated but costly: the restart link returns straight away while the old server keeps answering for several more seconds. Checking whether a change took effect immediately after clicking it reads the dying process and shows the old behaviour — which briefly looked like the fix hadn't worked at all. Wait for the new process before believing a post-restart page.*

- **Max day margin: a ceiling on how much margin can be tied up at once.** Switch it on in the strategy settings (Risk group), give it a dollar figure, and the strategy refuses any new trade that would push total committed margin past it.   *Dev note: the margin figures are never typed into the strategy — they're read back out of NinjaTrader at the moment of every single trade attempt, because brokers move day margins around (usually before a holiday or an expiry) and a number hard-coded today is wrong next month. Tested both ways — silent on today's healthy positions, and on a replay of the incident it flagged both naked trades while correctly clearing a third position on the same account, same second, that did have a stop.*
- **The two trades that went unrecorded have been reconstructed** and added to the trade history, so realized P&L, the template ledger and the ML data are no longer missing them: −$750.00 on template 7 and −$587.50 on template 13.
  *Dev note: every figure was sourced rather than guessed — prices and times from NinjaTrader's log, the data series and template from the frozen export files, and the strategy-setting columns computed from the template table in code and then checked against a live trade on the same template today, which matched exactly. Four columns were deliberately left blank: they're live volatility readings taken at entry on a Renko-1 and a HeikenAshi-1 chart, and nothing comparable exists to derive them from. A blank can be filtered out later; an invented number would look like a real measurement forever.*

Same failure, third account — which settles that this was a recurring hole and not a one-off.
- **Second alarm added: "the strategy has gone quiet."** If temalimit stops writing for 15 minutes while NinjaTrader is still running and still receiving prices, your phone gets a separate alert. This is the one that mattered most today.
  *Dev note: the trigger for adding it is the stall described above — the strategy showed **Enabled** in the Control Center while not actually running for over two and a half hours. "Enabled" and "executing" turn out to be different things, and nothing was checking the second one. The alarm deliberately requires two conditions at once: temalimit silent AND something else demonstrably alive. That pairing is what keeps it quiet on weekends and holidays without needing to know a single thing about market hours — when the market closes, everything goes quiet together, and that isn't a fault. If the check that looks for NinjaTrader can't answer, it assumes it's running, so a broken check fails loudly instead of quietly switching itself off.*

### 🎨 UI
- **The Live dashboard's jump bar now has a button for each chart type.** After the five section buttons there's a divider and then one button per chart type on the board — Line Break, Tick, Range, Volume, Renko, Minute, Heiken Ashi — each jumping straight to that card. Where two cards sit side by side it was always crediting the right-hand one, which on a normal-width window meant the entire left column of chart-type cards could never be highlighted at all — three of the seven new buttons would have been permanently dark.   *Dev note: the second one is the more interesting failure. Neither bug was new; both were invisible while the bar only listed five stacked sections, and both became unmissable the moment it listed seven cards laid out in a grid. Worth noting as the general shape: adding the feature didn't break these, it revealed them.*

### 🧭 Known issues
- **The two trades that triggered all this were not rescued by the fix** and had to be flattened by hand. Both were ES shorts — one entered at 7499.50 with a refused stop at 7507.25, the other at 7502.25 with a refused stop at 7511.75, together about **−$1,340** realized. It was not made, deliberately: the alarm trusts only confirmed position changes, never an instruction that might not complete.
- The underlying trigger — the strategy's price reading briefly lagging the broker's during a fast spike — is mitigated here, not eliminated. The new guard catches the consequence; it doesn't make the two readings agree.

## Patch 2026-07-19 — "Regime Change"

The market's July 16–17 selloff set off two data alarms. This patch teaches the alarms the difference between "the market changed" and "the data broke," fixes a stat freeze in background tabs, and adds a hardware monitor for the machine itself.

- **The time-range filter from the Live dashboard now exists on all four pages.** Models, Ops, and Trend each get the same 1D/2D/3D/5D/All picker Live has, counting days the way the trading desk does — a "day" starts at 3pm California, not midnight. On Models and Trend it filters the sample- and prediction-derived panels (counts, label mixes, recent activity); the model-health tables intentionally ignore it, since "is training healthy" is a now question, not a last-Tuesday question — each picker's tooltip spells out which panels it covers. Confirmed working on a phone-size screen.
  *Dev note: one subtlety worth knowing when numbers look "wrong": 3D and 1D can legitimately show the same count. Friday's samples logged at 1pm belong to the trading day that started Thursday 3pm — so a 3-day window anchored at Friday 3pm excludes them. The math is right; the anchor is just stricter than a calendar. On Trend, the staleness and anomaly alarms deliberately keep reading unfiltered data — whether the system is alive must not depend on which view you picked.*

It only pings the phone on *lasting* changes — never brief spikes — and every alert ends with a plain verdict: "64GB likely safe," "64GB borderline," or "keep 128GB." This is the one moment each day the training data actually changes, so it's the only moment a fresh check can find something new.
  *Dev note: prompted by a fair question — "if the Run button is always clickable, how would I know when to actually click it?" The clickable state was never meant to mean "due for a run"; now there's an actual daily run to rely on, and the panel's description spells out the difference.*

All four sections' splits check out exactly against the total sample count (394 live / 9,375 shadow / 9,769 total).

One caveat carried over from the existing per-market tables: clicking the "Count" column header to sort no longer sorts cleanly by the big number when a row has this extra live/shadow line underneath it — a pre-existing quirk this reuses rather than introduces, flagged for a future fix.*

### 🐛 Fixed
- **The trend robot's dashboard said "Idle — market likely closed" while all six of its markets were wide awake.** The banner at the top of that page is supposed to answer one question: is the system alive? It was answering a different one. It watched for the robot *asking the prediction service for an opinion* — and the robot only asks when a setup actually forms or nearly forms. On a thin Sunday evening that can easily be an hour of silence with everything running perfectly. Worse, oil is currently the only market that has ever produced such a request, so the entire "is anything alive" verdict rested on that one market happening to find a setup within twenty minutes. It hadn't for thirty-eight, so the page declared the market closed. The banner now watches the heartbeat each robot writes on every single bar it processes, whether or not anything interesting happened — a direct signal of "I'm running and my data is flowing." All six markets immediately showed up alive and the banner flipped to Healthy. Two supporting adjustments came with it: the "this one market has gone quiet" alarm moved to the same heartbeat signal, and its patience doubled from one hour to two, because how often a bar closes depends entirely on how busy the market is — at the moment of the fix the euro had ticked seven minutes ago while the Nasdaq's last bar was sixty-four minutes old, and both were perfectly healthy. The old one-hour limit would have cried wolf over the Nasdaq. Retired accounts also leave heartbeat files lying around forever, so anything untouched for a day is now ignored rather than mourned indefinitely.
  *Dev note: this is a cousin of the freshness-gate bugs above, but the lesson is its own — the alarm was measuring the wrong thing rather than measuring the right thing badly, and no threshold tuning would have saved it. Worth noting the near-miss during the fix: had the liveness signal been switched to heartbeats while leaving the per-market quiet alarm on the old signal, a harmless "Idle" would have become a permanent false warning the moment oil went quiet — a strictly worse outcome than the bug being fixed. When a banner has two alarms reading the same source, they move together or not at all.*
- **On the Live dashboard, picking a longer time range made chart-type cards disappear rather than appear.** Ask for the last 7 days and the Volume, Range and Heiken Ashi cards simply weren't there — even though those chart types had 27, 11 and 4 trades inside the week you asked for. The cards were being built correctly and then hidden on the way out the door. The page has a separate, deliberate rule that retires a card once its chart type hasn't traded in 24 hours, so the board doesn't accumulate ghosts of setups no longer in use. That rule was being applied on top of *every* range, including ranges explicitly asking to look further back — so any chart type quiet for a day vanished from a 7-day view that existed precisely to show it. Weekends made it worst: by Sunday almost everything is more than a day stale, so the longer ranges emptied out exactly when you'd reach for them. Now, if you name a range, the range decides what you see, full stop; the 24-hour retirement rule applies only to the "All" view, where nothing else would ever remove a stale card. Checked all three ways: 7-day shows all seven chart types, 1-day shows the four that actually traded, and "All" still correctly retires the three quiet ones.
  *Dev note: this is the same mistake as the data-series breakdown fix from earlier — a freshness gate and a date filter answering two different questions with one flag. Worth remembering the shape of it: "has this traded recently" is an absolute question about now, "does this have trades in the window" is a relative one about the range you picked, and stacking the first on top of the second silently subtracts from what you asked for. If a third place ever gates on the 24-hour window, check it against a weekend before trusting it.*
- **Two stray keystrokes quietly froze the entire robot codebase — every fix made all evening was written down but never actually running.** A comment somewhere in the chart-data helper began with a single backslash instead of the two slashes that start a comment, and the identical slip sat in the main strategy file. NinjaTrader treats one bad character anywhere as a reason to refuse the whole batch, so it kept running the last version that *did* compile. Nothing looked wrong — the robots kept placing orders normally all night — but they were trading yesterday's logic. Every repair from this evening (the rotation-mode memory fix, hand-closed-trade logging, the trailing-stop hardening, the stop-fill correction) sat on disk doing nothing. Both characters fixed and every file swept for the same slip.
  *Dev note: worth internalizing how quiet this failure is — the strategies keep trading, the log files say nothing, and the only clue in the entire system is a red row in the editor window that nobody has open. Treat "I edited the strategy" and "the strategy changed" as two separate claims from now on.*
- **Killing the prediction service didn't kill the work it had started, so ten of sixteen processor cores stayed busy on nothing.** Model training runs in separate helper processes. The scripts that stop and restart the service identified what to shut down by looking for its startup command — which those helpers don't share — so they were skipped, kept running with no parent, and nothing was ever going to clean them up. Two of them had been chewing through ~10 cores since the evening's earlier restarts. The freshly restarted service was left fighting for scraps: a basic health check went from a fifteenth of a second to nearly five seconds, which is what made the manual retrain button appear frozen (it shows nothing at all until the retrain fully finishes, so "very slow" and "hung" look identical). Both scripts now follow the family tree down and shut the helpers down too.
  *Dev note: with the strays cleared, a manual retrain finished in about three minutes and trained nine of ten eligible markets. Checked before and after: still zero markets cleared the quality bar to gate entries, so the robots continue trading on their plain signal exactly as before.*
- **A model now has to prove it beats a coin flip — and prove it on enough evidence to count — before it's allowed to recommend anything.** Until tonight, a market was promoted from "just watching" to "the model may now offer opinions" on two conditions: a saved model file exists, and at least one trade has completed. Nothing anywhere asked whether the model was any *good*. The two exit models trained tonight scored dead-even with a coin toss and sailed through that check — they'd have started recommending exits the instant the exit system was switched on. Two gates now stand in the way. First, an accuracy floor: score better than 0.55 or stay in watch-only mode. Second, a minimum-evidence floor: the model must have seen at least 100 examples of the rarer of the two outcomes. A model whose record can't be read at all — missing or corrupted results file — is treated as failing rather than given the benefit of the doubt, on the principle that an unverifiable model is not a trustworthy one. Both of tonight's models correctly dropped back to watch-only the moment the change went live, and the log now names every reason a market was held back rather than just the first one it hit.
  *Dev note: the second gate exists because the first one can't stand alone. Tonight's models scored 0.50 — but that was luck cutting in our favour. A model that has seen three examples of "exit" can just as easily score 0.83 by coincidence, and a lone accuracy floor would wave it straight through. The evidence bar was first set at 25 on a rule of thumb, then corrected to 100 by an actual experiment run the same night: holding everything else fixed and varying only how many rare-case examples the model sees, the score climbs 0.44 → 0.47 → 0.52 → 0.53 → 0.56 as that count goes 2 → 5 → 20 → 50 → 100. Fifty examples still measured a coin flip, so 25 would have let noise through. The same experiment carries a sharper lesson: at two examples the score wobbles by ±0.27 between random reruns, meaning the accuracy floor is barely measuring anything down there — the evidence floor is the gate actually doing the work, and the accuracy bar is the second line of defence. 100 also matches the bar the training step itself now uses, so the two stages finally agree. Both checks read from the results file on disk rather than from memory, because the in-memory copy still holds the previous run's numbers at the moment the decision is made — an ordering trap that would have silently scored the wrong model. Verified with 21 checks, including the specific case the accuracy floor alone would miss.*
- **The first two exit-timing models ever trained finished successfully — and are worth exactly nothing, which is useful to know.** Both long-running training jobs completed and saved their models before being shut down. But look at what they learned from: the ES model saw 70,218 examples of "keep holding" and precisely **one** example of "exit now"; the NQ model saw 37,215 versus **three**. A student shown one counter-example cannot learn the difference, and the scores confirm it — both models perform at exactly coin-flip accuracy. They're stamped "trained OK" all the same.
  *Dev note: the trap worth knowing about is that "has a saved model file" is currently enough to promote a market to recommendation mode — there's no check that the model is better than a coin flip. That's harmless right now because the exit system is globally switched off and only collecting data, and the gate for letting a model actually close real trades is properly strict (it needs 150 completed trades, a genuinely better-than-chance score, and four weeks of varied data — these have 9 and 21 trades, coin-flip scores, and two weeks). But an accuracy floor belongs in that promotion check before the exit system is ever switched on.*
- **Winners-First on NQ picked template 1 — the one thing it should never do.** The new Losers-First/Winners-First rotation modes are supposed to open on the biggest loser or biggest winner in the profit ledger, but a fresh NQ instance in Winners-First mode calmly started on template 1, which wasn't even in its ledger. Three separate problems, found by pulling that one thread:
  1. **The ranking ran before the strategy knew which account it was on.** The "start at the top of the ranking" step runs during startup, early enough that the account name can still be unknown — and the ledger's filename contains the account name. Wrong name → file not found → empty ranking → keep the default template, silently. The strategy now re-checks the ranking a second time at the moment it goes live (when the account is guaranteed known), and only then locks in its opening pick — with a printed line saying what it changed and why.
  2. **The profit ledger was filed by the wrong things — blending chart types, while splitting accounts.** It was kept per account + market, so a 1000-volume NQ instance and a Renko NQ instance on the same account shared one pot of "how has template 38 done?" — even though the same template behaves completely differently on different chart types. Meanwhile every account kept its *own* pot, so a freshly-created account started with a blank record and nothing to rank, even when an identically-configured account had months of history on the exact same market and chart type. Both are now the other way round: the ledger is filed **per market + chart type, with all accounts pooled** — how a template performs on NQ 1000-volume is a fact about the template and the chart, not about whose account placed the order. Mode 3's separate "already tried" checklist is filed exactly the same way, for the same reason — whether template 12 has been tried on NQ 1000-volume is a fact about the chart, not about the account, so two accounts on the same market and chart no longer each re-test the full set of 40 from scratch. All 27 ledgers were rebuilt from the master trade log, which always recorded the chart type. Dropping the account from the key also brought back 207 older trades that had to be discarded earlier in the day — they predate the trade log recording accounts at all — so the rankings now count **343 of 343 trades instead of 136**. Old files archived, not deleted. For the curious: on NQ 1000-volume the all-time board is T39 +$6,670, T40 +$2,210, T26 +$1,275, T38 +$1,075, T35 +$50 — thin samples, but that's what Winners-First will now actually walk.
  3. **Trades closed by hand never made it into the ledger at all.** The ledger only heard about exits the strategy itself ordered. Manual closes and crash-recovery bookings — which yesterday's fixes taught the *trade log* to record — skipped the ledger entirely, so a template's ranking could drift from its true record (one NQ template was short $367.50 from a single evening). All three paths now write the ledger, with the same photo-finish handling the trade log already has: if a "manual close" turns out to have been the strategy's own stop, the ledger is corrected to the true fill price rather than counted twice.
  *Dev note: the T1 sighting that kicked all this off actually had two different causes in sequence — the startup-timing bug above, and then, after that was fixed, a brand-new account with a genuinely empty ledger, which is the designed fallback rather than a bug. The second one is what made the account-pooling decision obvious. Pooling the mode 3 checklist was initially argued against on the grounds that instances would starve each other of untried templates; a 400-run simulation of the actual rotation code showed that's simply not how it fails — when the checklist empties, every instance falls back to plain round-robin and keeps trading. What the simulation did find, with two accounts on one market and chart: pooled covers all 40 templates in 58 trades versus 89, and cuts duplicated testing from ~38% to ~24%, at the cost of the two instances more often holding the same template at the same moment (about 31% of trades versus 10%) — because "lowest untried" is deterministic, so siblings reading one list land on the same answer. That last part is a position-correlation trade-off worth knowing about, not a data problem. Verified the whole file still parses clean after every one of today's edits (zero syntax errors via a standalone compiler pass), since a single typo anywhere under bin/Custom silently freezes all strategies on stale code.*
- **The pullback-state file writer now waits out virus scanners instead of giving up.** The strategy exports a small dashboard file after every bar close, and occasionally logged "unable to write after 5 attempts — file in use by another process." Investigation showed the writer itself is fine; the trouble is that the safe atomic-swap technique it uses makes Windows treat every update as a brand-new file, which sends Defender and the search indexer to re-scan it — and their brief lock sometimes outlasted the writer's old tenth-of-a-second patience. The writer now retries with growing pauses for up to about 1.3 seconds. Missing a write was always harmless (the next bar rewrites everything), so this just quiets the noise and keeps the dashboard fresher.
- **A trade closed by the strategy's own stop-loss got recorded as if a human had closed it — now the record fixes itself.** Yesterday's "manual flatten" safety net logs any trade the account closes out from under the strategy, using the current market price since the true fill isn't known yet. Tonight it caught its first false positive: an NQ trailing stop filled at 20:47:20, but the "account is flat" news reached the strategy a split second before the news of its own stop filling did, so the trade went into the books as a manual close at an estimated price — off by less than a point, and filed under the wrong reason. The safety net still fires exactly as before, but now, if the strategy's own exit fill shows up moments later, it goes back and corrects the record: right reason ("Stop loss"), right price, right profit number, and the daily P&L total is adjusted by the difference. Trades a human genuinely closes are untouched — there's no strategy fill coming, so nothing rewrites them.
- **The chart-data helper no longer prints its routine "late callback, discarding" housekeeping note to the NinjaTrader output window.** It was working as designed — a slow data fetch answered after a faster retry already won — but it read like an error and cluttered the log. The behavior is unchanged; only the noise is gone.
- **The exit-model's first real training run asked the machine for 311 gigabytes of memory — and kept asking, every few minutes, all evening.** In a way this was a success story gone wrong: yesterday's exit-model repair worked, ES 3LineBreak trades finally closed on Sunday's session, and that market's exit model became eligible to train for the very first time. But its training data is unusual — nearly 70,000 bar-by-bar snapshots spread across just nine trades, some of which sat open for tens of thousands of bars — and the training code packed up, for *every single bar*, a complete copy of everything that came before it in that trade. The bill came to 311GB in one allocation. Windows spent minutes at a time gamely swelling its swap file toward that number before giving up, and during those minutes the whole prediction service went unresponsive — that's what the "ML POST FAILED" timeouts in NinjaTrader were, along with a watchdog-forced restart at 19:08. Failing didn't stop it: 200 samples later (about three minutes at Sunday-evening pace) it tried again, forever. Training examples are now capped at each trade's last 128 bars — which is all the model ever actually read; its architecture makes its call from the end of the sequence — bringing the worst case from 311GB down to about a third of one. Live predictions trim to the same window so the model sees in production exactly what it saw in school.
  *Dev note: the three sibling 3LineBreak markets (RTY, NQ, YM) were queued up to hit the identical wall as soon as their trades started closing — this fix lands before any of them got there. Separately: the afternoon's scattered timeout bursts around 2:43–4:37pm were nothing of the sort — each one lines up exactly with someone clicking the dashboard's own Restart button, and the hidden duplicate SYSTEM-level watchdog task found during the same investigation (it was double-checking the service every minute and defeating the "fail twice before restarting" safety) has been deleted by hand.*
- **On the Models page, the floating navigation pills no longer sit on top of a card's title.** Clicking a pill to jump to a section could land you with the top of that card hidden underneath the pill bar — especially on narrower windows, where the header wraps onto two lines and gets taller than the fixed amount of headroom the page reserved for it. The page now measures the header's actual height (and re-measures whenever the window is resized) and scrolls sections just past it. The pill bar also got a more opaque, blurred background so card content no longer shows through it.
- **Template rotation mode 3 no longer forgets its place after a restart.** Re-enabling the strategies after tonight's NinjaTrader crash sent every "Unused Only" instance back to template 1, even though each one had faithfully saved the template it was on. That reset was actually written into the code on purpose — "on startup, always jump to the lowest template that hasn't traded yet" — but a template only earns its "has traded" mark when a trade *closes*, and none of these instances had closed one yet, so "lowest untried" always meant template 1. Mode 3 now picks up exactly where its save file says it left off, the same way the other rotation modes always have; the jump-to-lowest-untried still happens where it belongs, after each completed trade.
  *Dev note: the crash was a red herring — nothing was lost, the save files were intact and correct. A clean restart on any day would have done the same thing. The Losers-First/Winners-First modes (4/5) still deliberately re-rank from the profit ledger on startup: their "saved place" is the ranking itself, not a template number.*
- **Trades you close by hand now show up on the Live dashboard instead of vanishing.** Until today, the trade log only heard about exits the strategy itself ordered — stop losses, targets, timed exits. Close a position from NinjaTrader's own buttons (or have the platform force-close one), and the trade evaporated: no row in the trade file, nothing on the dashboard, as if it never happened. Two real trades disappeared this way tonight after a NinjaTrader crash forced a manual cleanup. The strategy now covers both ways this happens: if it's running and notices the account went flat underneath it, it writes the trade down on the spot (marked "ManualFlatten", priced at the market's level that second); and when an instance starts up and finds a leftover "open trade" note from a session that died mid-position, it books that orphan as a manual close too — so tonight's stranded SimRenko trade will file itself the moment that instance is re-enabled.

  *Dev note: the error was being caught and printed by a safety net wrapper, which is exactly why it looked cosmetic — one tidy line in the output and no crash. But the wrapper sits around the whole startup routine, and the throw happened near the top of it, so everything below was skipped. The one-word tell was in the message itself: "between 0 and -1" is the platform saying it has minus-one bars. Also fixed the debug-print helper, which had the same flaw and would have thrown while trying to report the original error. - **The Active Templates card listed templates from configs that aren't running anymore, and from days ago.** Each strategy instance leaves a small "which template am I on" note on disk — but the note's filename includes the chart type, so switching an instance to a new chart type left the old note behind forever, and the card kept showing it as if it were live. Confirmed the cutoff behaves right at the boundary (currently 16:05 PT, cutoff correctly lands at today's 15:00 PT) — before /restart the card had 52 rows across two trading days, after it dropped to 16, all from the current one.*

The strategy now writes that note once at startup for fixed-template instances too, and the card exempts them from the "rotated today" rule — a template that's pinned on purpose isn't stale, it's just pinned.
  *Dev note: the note file gained a real second field (it was a dead always-zero slot) carrying the template mode, encoded negatively so the thousands of old files with a literal zero there can't be confused for fixed-mode instances — misreading them would have resurrected every stale row the earlier fix just hid, permanently, since dead files never get rewritten. Sit in a position for the afternoon, or leave an entry order working, and the instance writes nothing for hours — then vanishes from the card at the 3pm day boundary while it's arguably at its most alive. The strategy now re-stamps its note file every 30 minutes no matter what it's doing, so a running instance can never age off the card.
  *Dev note: heartbeat runs before the rotation-eligibility check on purpose, since that check is exactly what was starving the saves. The timestamp updates even if the disk write fails, so a persistent write error paces itself at one retry per 30 minutes instead of one per bar. Running instances pick this up on their next restart.*
- **The Models dashboard kept jumping to a different card on phones.** Try to read the Active Templates card on mobile and within a couple of seconds you'd be looking at something else. ### 🔧 Under the hood
- **The exit robot's retraining had quietly turned into a permanent background job pinning the machine.** Two of the fastest-logging chart types (ES and NQ 3-Line-Break) were retraining essentially non-stop — each retrain took over an hour, but a new one came due every ~4 minutes, so the moment one finished the next started. The rule was "retrain every 200 new observations," which is sensible for a small group but ignores that the work itself grows with the group's *entire* history: at 70,000 observations, those 200 new rows were triggering an hour of work. Retraining is now spaced out in proportion to how much history a group has. Small groups are untouched and still retrain every 200 observations; the big 3-Line-Break groups go from every ~4 minutes to roughly hourly. Retraining also no longer grabs every CPU core on the box, so it can't crowd out the live trading decisions it shares the machine with.
  *Dev note: this is the same pair of groups behind last night's 311GB memory blowup — that fix stopped the crash but left the underlying churn, which only became visible as sustained CPU load once training could actually complete. Worth stating plainly: these groups have logged tens of thousands of observations from just 9 and 20 finished trades, and none of their models are being used for live decisions yet, so slowing this down costs nothing real today. The restart baseline is seeded from each group's existing count, so restarting the service no longer sets off a simultaneous retrain of every large group.*

- **The exit robot was burning hours of CPU training on data it could not possibly learn from.** Follow-up to the retraining-cadence item above, which turned out to be treating a symptom. Spacing the retrains further apart helped and was working exactly as built — but the machine went straight back to full load, and the reason was in the finished models themselves: they had learned *nothing*. One chart type had **2 "exit" examples against 90,627 "hold" examples**; another had 5 against 39,665; two more had zero exits at all. A model trained on that scores 0.5000 — a coin flip, the exact number you'd get by guessing. The permission check only asked "has this group seen at least one of each outcome?", and a single row answered yes. It now requires real evidence: at least 50 examples of the rarer outcome, *and* at least 1% of the group. All six expensive chart types stop training immediately and resume on their own if they ever accumulate enough genuine variety. Nothing of value is lost — none of these models fed a live trading decision.
  *Dev note: two lessons worth writing down. First, an honest correction — the earlier estimate of how fast these groups log data was measured before the Sunday open and was 6x too low; the real rate is ~351 rows/minute. Second, and more useful: spacing retrains proportionally to group size **cannot** fix this kind of overload, no matter how the dial is set. Training cost grows at exactly the same rate as the gap between retrains, so the group size cancels out and the dial only picks a fixed level of oversubscription — at the current setting, retraining takes 10.6x longer than the gap it's given; even the most aggressive sane setting still lands at 1.1x. The dial stays in place as cheap insurance, but the real fix was refusing to start.*

- **Three weeks of exit-training data was missing the one thing that mattered — and nothing was checking.** Asked whether any data was actually poisoned, the answer turned out to be yes, though not in the obvious way. Every trade entered before July 18 — **613 of 644** — never recorded the moment it closed. Each one is a long transcript of "still holding, still holding, still holding" that simply stops, with the exit silently absent. That's the fingerprint of the bug fixed on July 18, and this data is the proof it worked: zero trades logged an exit before that date, and 20 of 28 have since. Those rows aren't false — the trade genuinely was being held during those bars — but with the ending cut off they teach the robot that positions never close. All 613 trades (185,032 rows) are now moved into an archive folder, not deleted, and verified row-for-row: nothing lost, and **not one genuine "exit" record was moved** — all 20 stayed put. A new **Exit-label integrity** check now runs alongside the other data alarms and fails if any *closed* trade is missing its exit, with trades still running excluded so an open position never trips it.
  *Dev note: worth being clear that the archiving fixes nothing by itself, which is why it wasn't rushed. Modelling the purge first showed that not a single chart type becomes trainable afterward — no group anywhere has more than 5 exit records total, against a floor of 50. The value is preventing bad rows from re-entering later and keeping the evidence. Also checked and cleared: no scrambled columns, no data logged under the wrong market, no duplicated rows, and no double-counting (the ~5 rows/second is genuine bar formation, not a logging loop).*
- **The new check found a live problem within a minute of being switched on.** Five trades from July 19 — *after* the fix — closed without recording their exit, including one with 18,175 rows. All five had gone quiet for 3–4½ hours while their chart types kept logging, so they're definitely closed rather than still open. So the July 18 fix is roughly 80% effective, not complete.
  *Dev note: the cause is genuinely not established yet and shouldn't be guessed at. One lead: three of the five went quiet within a couple of minutes of each other, shortly after a service restart, which would fit their exit records being lost while the service was unavailable. Against that theory, NinjaTrader's own logs record no failed sends at all that day, and the other two don't line up with any restart. Next step is checking whether the exit record is sent fire-and-forget with no retry — but that's a look, not a conclusion.*

- **A closing trade now refuses to lose its exit record.** The cause of the missing exits from the item above turned out to be the delivery, not the logging. When a trade closes, the strategy sends one final "here's how it ended" message — and it was sent the same throwaway way as the thousands of routine "still holding" updates: fire once, and if it fails, shrug. A trade sends about 2,000 of the routine ones and exactly **one** of the important one. Miss that single send — because the service was restarting, or just busy past a 2.5-second cutoff — and that trade's ending is gone for good. It now retries four times, and if the service still isn't answering, writes the message to a file on disk and keeps re-sending it every minute until it lands. The routine updates deliberately keep the old cheap behaviour; they're redundant by design.
  *Dev note: an honest retraction from the earlier entry — I'd said NinjaTrader's logs showed no failed sends, treating that as evidence against this theory. That check was wrong twice over: those messages are rate-limited to one per 30 seconds, and they go to the Output window rather than the log files I searched. Also worth recording a trap for next time: the syntax checker was first run with its output pointed at a null path, which makes the compiler quit before reading a single line — producing a confident "no errors" that meant nothing. The real check found the file clean, and NinjaTrader rebuilt it the same minute.*
- **Studied how much data the exit robot actually needs — and the study contradicted the plan.** The assumption all evening was that the lopsidedness of the data (roughly 2,400 "holding" rows per "exit" row) was what stopped the robot learning. Tested directly, that's **false**: holding everything else equal and varying the lopsidedness across the entire range, the model scored identically every time. What actually decides it is the plain number of *finished trades* — below about 50 the score is indistinguishable from a coin toss, and at 2–5 it swings so wildly that a "good" score is pure luck. That precisely explains the 0.5000 seen earlier: it wasn't a measurement, it was noise.
  *Dev note: this changes the recommendation and corrects two of my own settings from earlier tonight. Trimming the data is still worth doing — capping each trade at its final 128 updates removes 96.5% of the rows and turns a two-hour retrain into minutes — but the justification is **cost, not accuracy**. Meanwhile the 1%-minimum rule I added is the wrong shape and should be relaxed, since it would block genuinely usable data that merely sits in a large pile; and the 50-example minimum is, if anything, too generous. The real bottleneck is simply that these chart types have completed 1–6 trades each, and no amount of clever sampling invents more.*

### 🧭 Known issues
- Two drift alarms remain honestly red (NQ 60-Range labels; several trend-geometry inputs): the market genuinely looks different after July 16–17. Expected to clear as new data accumulates — still red in a week means it's a real signal worth acting on.
- The 3-Line-Break and Renko chart types log tens of thousands of "keep holding" rows per finished trade and almost no "exit" rows, because a single trade can span that many bars. Until that shape changes, those groups will keep skipping exit-model training by design. Worth revisiting how exit samples are recorded for these bar types rather than raising the floors.
- Five July-19 trades closed without recording an exit (see above). Until that's understood, the exit-label check will keep failing — correctly. It should not be silenced to make the dashboard green.

`temalimit` doesn't use one fixed set of rules all day. It has 40 different settings profiles ("templates"), numbered 1 (strictest, fewest trades) to 40 (loosest, most trades), and it rotates through them over time. The picture below shows the rotation logic:

![Template Mode 3 rotation wireframe](wireframes/mode3_wireframe.svg)

In plain terms:
- Every time an order is placed, there's a countdown. If the profile that placed the order has won a trade before, it gets more time to fill (69 minutes); if not, it gets less (about 34 minutes).
- If the order fills in time, the strategy trades it out normally.
- If it doesn't fill in time, the strategy gives up on that profile and moves to the next one in line.
- Once a trade is open, the strategy never switches profiles until that trade is closed.
- After a **win**, it keeps using the same profile. After a **loss**, it steps back one profile (gets a little stricter), unless it was already on the strictest one, in which case it resets to profile 1.
